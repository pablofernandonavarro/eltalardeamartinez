## Project Overview

**El Talar de Martínez** is a residential complex management system for a gated community in Argentina. It handles property/unit/resident administration, pool access via QR codes, expense tracking, and SUM (multipurpose hall) reservations.

## Architecture

### Stack
- **Laravel 12** + **Livewire 3** — traditional PHP class + Blade view pairs (Volt is installed but not used; do not use single-file Volt components)
- **Livewire Flux** — primary UI component library; use `<flux:button>`, `<flux:input>`, `<flux:dropdown>`, `<flux:icon.*>`, etc. for all UI elements
- **Alpine.js** — lightweight interactivity within Blade templates
- **Laravel Fortify** for authentication (2FA supported)
- **TailwindCSS v4** + **Vite**
- **SQLite** (local dev) — configured in `.env`
- **PestPHP v3** for tests
- **Maatwebsite Excel** for imports and exports, **DomPDF** for PDF generation
- **FullCalendar** for the SUM reservations calendar view

### Livewire Component Pattern
Every feature is a Livewire component — traditional controllers are minimal. Each component has two files:
- Class: `app/Livewire/{Area}/{Name}.php`
- View: `resources/views/livewire/{area}/{name}.blade.php`

Components are organized into four areas: `Admin/`, `Resident/`, `Banero/`, `Settings/`.

### Routes
| File | Middleware | Purpose |
|------|-----------|---------|
| `routes/web.php` | `auth`, `approved` | Public landing + resident-facing pages |
| `routes/admin.php` | `auth`, `admin` | Admin CRUD for all entities |
| `routes/banero.php` | `auth`, `banero` | Pool attendant views |
| `routes/api.php` | — | SUM calendar events endpoint |

Custom middleware: `EnsureUserIsAdmin`, `EnsureUserIsApproved`, `EnsureUserIsBanero`.

### Roles
Defined in `App\Role` (namespace `App`, not `App\Enums`): `Admin`, `Banero`, `Propietario`, `Inquilino`, `Residente`.

Users also require a non-null `approved_at` timestamp to access the app. New registrations go through an admin approval step.

### Key Domain Models
- **User ↔ Resident**: A `User` is an auth account; a `Resident` is a family member (minor or dependent). They can be linked (`user_id` on `residents`). QR tokens exist on both.
- **Complex → Building → Unit → UnitUser**: The property hierarchy. `UnitUser` is the pivot linking users to units, with `is_owner` and `is_responsible` flags.
- **PoolEntry**: Logs each pool access with entry/exit timestamps. Pool access is controlled by `PoolRule` records (date ranges, guest quotas, capacity).
- **PoolDayPass**: A daily guest permit generated by residents. Contains a UUID `token` that encodes the QR code and links to the approved `PoolGuest` list for that day.
- **SumReservation**: Has an approval workflow — states flow through `SumReservationStatus`: `pending` → `approved`/`rejected`/`cancelled`/`completed`. Each transition stores who performed it and when (e.g. `approved_by`, `approved_at`).
- **SumPayment**: Tracks payment for a `SumReservation`. Status via `SumPaymentStatus`: `pending` → `paid`/`cancelled`. Has `payment_method` (cash/transfer/card/online).
- **Expense → ExpenseDetail → Payment**: Expenses are allocated per unit via `ExpenseDetail`; residents see and pay their own `Payment` records.
- **SystemRule**: Dynamic business rules for unit occupancy and pool guest limits. Stores `conditions` and `limits` as JSON arrays, scoped by `building_ids`/`complex_ids`, with a `priority` field (higher = evaluated first) and optional `valid_from`/`valid_to` date range.

### QR Code System
There are four distinct QR token types, all handled by `Banero\Pools\Scanner`:
1. **User QR** — UUID stored in `users.qr_token`; scanning toggles entry/exit for that user
2. **Resident QR** — UUID stored in `residents.qr_token`; for 15+ year-old residents who can enter independently
3. **Day-Pass QR** — UUID stored in `pool_day_passes.token`; generated daily by residents via `Resident\MyPoolQrUnified`; presents the approved guest list for confirmation at the gate
4. **Exit QR** — static token `POOL_EXIT_2024`; a posted sign at the exit that opens a list of all open entries to select for checkout

### Service Layer
Business logic lives in `app/Services/`, not in Livewire components or controllers:
- **`PoolAccessService`** — `registerEntry()` / `registerResidentEntry()` create pool entries after validating all access rules (pool enabled, user eligibility, owner-only mode, daily limits, weekly/monthly guest budgets)
- **`RuleEvaluationService`** — evaluates `SystemRule` records in priority order; used for unit occupancy checks and pool guest rule validation
- **`ExpenseService`** — expense calculation and distribution across units
- **`PaymentService`** — payment processing and status updates

### Events & Listeners
Domain events are in `app/Events/` with listeners in `app/Listeners/`. Key events: `ExpenseCreated`, `ExpenseDetailsGenerated`, `ExpensePaid`, `PaymentRegistered`, `PoolEntryRegistered`, `PoolRuleUpdated`, `ResponsibleChanged`. The queue worker must be running for async listeners.

Model observers in `app/Observers/`: `ExpenseObserver`, `UnitUserObserver`.

### Artisan Commands
20+ custom commands in `app/Console/Commands/` for data management: generating QR tokens, syncing residents↔users, importing from Excel, cleaning up pool entries. Use `php artisan list` to see all.

### Enums
Status/type fields use PHP 8.1 backed enums. Older enums live in `app/` (namespace `App`): `Role`, `PoolStatus`, `ExpenseType`, `ExpenseStatus`, `PaymentStatus`. Newer SUM-related enums live in `app/Enums/` (namespace `App\Enums`): `SumReservationStatus`, `SumPaymentStatus`.

## Database

SQLite in development. Migrations are in `database/migrations/` (47 migrations). Most models use soft deletes. The `database/database.sqlite` file is gitignored — create it with `touch database/database.sqlite` then run `php artisan migrate`.

## Localization

The app uses `laravel-lang/common` for Spanish translations. Timezone is `America/Argentina/Buenos_Aires`.
